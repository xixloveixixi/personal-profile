# Agent 开发规则

## 研发 Harness 必读
在本项目执行任何开发任务前，必须先读取：

- `docs/dev-harness/stage-plan.md`
- `docs/dev-harness/progress-log.md`
- `docs/dev-harness/pitfalls.md`

## 工作流程

1. 先根据 `stage-plan.md` 判断当前阶段。
2. 编码前检查当前阶段的 Gate A-D。
3. 只推进一个可验证小闭环。
4. 修改代码前说明影响范围、依赖关系和验收方式。
5. 如果当前操作涉及目录结构变更或新增 package，先回看 Gate B/D 确认是否对齐设计。
6. 每个闭环完成后输出产出物检查（见下方），等待用户确认后再推进下一个闭环。
7. 完成后更新 `progress-log.md`。
8. 如果出现可复用踩坑，更新 `pitfalls.md`。
9. 如果编码过程中做了非平凡重构（新增 package、调整依赖方向），评估是否写入 `pitfalls.md`（重点记录"为什么要这样改"）。

## 闭环产出物检查

每个闭环完成后，AI 必须输出以下内容，等待用户确认后才能推进下一个闭环：

1. **文件变更清单**：列出新增/修改/删除的文件。
2. **验收结果**：curl 输出、测试结果或编译通过截图。
3. **当前 Gate 状态**：标记本闭环完成了哪项 Gate E 检查项。

## 节奏控制

- 一次对话最多推进 2 个 Day，推进完毕后必须暂停等待用户发起下一步。
- 如果用户主动要求连续推进，可以继续，但每个 Day 结束仍需输出产出物检查。

## 限制

- 不允许跳过阶段直接开发后续功能。
- 不允许在未确认接口和表结构前写业务后端。
- 不允许把博客迁移出 Notion。
- 不允许一次性大范围重构现有前端。
