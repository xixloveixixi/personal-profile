# 测试规范

## 当前阶段（V1 Stage 1）

项目尚未引入单元测试 / E2E 测试框架，质量门禁通过以下命令保证：

- `npm run type-check`：TypeScript 类型检查必须通过。
- `npm run lint`：ESLint（`eslint-config-next`）必须无错误。
- `npm run build`：Next.js 生产构建必须成功。

合入前必须本地通过以上三条；CI 也以这三条为最低门禁。

## 类型与契约

- 所有 Route Handler、`lib/` 封装的输入输出必须有显式 TS 类型。
- 来自外部（API、Notion、LLM）的数据必须经过 `zod` 校验，校验失败按 `05-API规范.md` 的错误码返回。

## 后续测试方案（按 SDD 第 14 章）

引入测试需先在 Harness Plan 中提案并写入 `stage-plan.md`，分阶段落地：

1. **Go 后端单元测试**：业务关键函数（计划生成、JWT 中间件、权限校验）。
2. **接口测试**：覆盖公开接口、私有接口（带 Token / 不带 Token）。
3. **前端测试**：核心交互（登录、新增目标、计划生成、复盘提交）的组件测试与简易 E2E。
4. **端到端验收**：按 SDD 14.4 手动跑通公开访问、登录、学习闭环、Chat 双模式。

## 限制

- **禁止**未经 Harness 评审就引入测试框架（vitest / jest / playwright），避免在 V1 阶段消耗精力。
- 单测落地时优先覆盖业务关键路径，不追求覆盖率指标。
