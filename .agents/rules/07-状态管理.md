# 状态管理规范（NON-NEGOTIABLE）

## 状态优先级

按以下顺序选择，避免无谓的客户端状态：

1. **Server Component 数据流**：能在服务端拿到的数据（Notion、Go API、本地内容）优先在 Server Component 内 `await` 获取并通过 props 下传。
2. **URL state**：可分享、可后退的状态（筛选、分页、tab）放在 URL search params。
3. **React 局部 state**：只与单组件相关的 UI 状态用 `useState` / `useReducer`。
4. **Zustand 全局 store**：跨组件共享的客户端状态（登录态、Chat 会话、学习工作台筛选等）。

## Zustand 使用规范

- 状态管理库统一使用 **Zustand**（`zustand` 包，需新增依赖）。
- 所有 store 文件必须放在 `lib/stores/` 目录下，**禁止**散落到 `app/` 或 `components/` 内。
- 每个业务模块单独一个 store 文件，文件名 kebab-case（如 `auth.ts`、`chat.ts`、`learning.ts`）。
- 每个模块必须暴露 `useXxxStore` Hook。
- Store 内只放纯数据与业务行为，**禁止**直接耦合 UI 组件、`react` ref、DOM。
- 引用 store 的组件必须是 Client Component（首行 `'use client'`）。

## 目录示例

```
lib/stores/
├── auth.ts         # useAuthStore：登录态、JWT、当前用户
├── chat.ts         # useChatStore：Chat 会话、当前消息流
└── learning.ts     # useLearningStore：私有工作台筛选、当前选中目标
```

## Store 写法约定

- 使用 `create<State>()(set => ({...}))` 形式定义。
- 选择器尽量细粒度，避免组件订阅整个 store 触发不必要 rerender：`const token = useAuthStore(s => s.token)`。
- 异步行为放在 store action 内，组件层只调用 action，不在 store 外组装多步业务流。

## 持久化

跨页面 / 刷新需要保留的状态使用 `zustand/middleware` 的 `persist`：

- 仅持久化必要字段（如 `token`、`user`），不要把所有 state 序列化。
- 敏感字段（Token）按需启用 `partialize`，并考虑后续切换到 httpOnly cookie。

## 主题与会话态

- 主题切换使用 `next-themes`（已集成），不再单独建主题 store。
- 登录态由 `useAuthStore` 持有，Token 也用于 `lib/` 下的私有接口请求。
