# 编码规范

## TypeScript 规范

- 所有代码必须使用 TypeScript，禁止使用 JavaScript（脚本除外，如 `scripts/*.js`）。
- 所有 `.ts` 和 `.tsx` 文件必须包含完整的类型定义。
- 禁止使用 `any` 类型（除非有明确理由并添加注释说明）。
- 接口、类型定义必须使用 PascalCase 命名。
- 组件 Props 必须定义清晰的 TypeScript 接口。
- 外部输入（API 入参、LLM 输出、Notion 数据）使用 `zod` 校验后再进入业务流。

## 命名规范

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 文件夹 / 路由段 | kebab-case | `review-weekly`、`portfolio` |
| React 组件文件 | PascalCase.tsx | `ChatWindow.tsx`、`ProjectCard.tsx` |
| 工具 / Hook 文件 | kebab-case.ts | `use-chat-stream.ts`、`format-date.ts` |
| 变量 / 函数 | camelCase | `userList`、`onSubmit` |
| 常量 | UPPER_CASE | `API_TIMEOUT` |
| 枚举及属性 | PascalCase / UPPER_CASE | `ApiMethod.GET` |
| 接口 / 类型 / 类 | PascalCase | `UserInfo`、`BaseResponse` |
| React 组件 | PascalCase | `UserProfile` |
| 自定义 Hook | use 开头 | `useAuthStore` |
| Zustand store hook | useXxxStore | `useLearningStore` |

## 业务函数命名

- **事件处理函数**：`onXxx`（如 `onBannerClick`、`onFormSubmit`），用于组件 props。
- **内部处理函数**：`handleXxx`（如 `handleDelete`、`handleSubmit`），用于组件内部逻辑。
- **数据请求函数**：见 `05-API规范.md`。

## Server / Client Component 边界（NON-NEGOTIABLE）

Next.js App Router 默认所有组件为 Server Component，遵循以下规则：

- 文件顶部声明 `'use client'` 才能成为 Client Component。
- Server Component 内禁止使用 `useState`、`useEffect`、`useRef`、浏览器 API（`window`、`document`、`localStorage`）。
- Server Component 优先用于数据获取（fetch / Notion / Go API），Client Component 只承担交互。
- 仅在需要交互、Hooks、动画、Zustand 时才转为 Client Component，避免无谓客户端化造成包体膨胀。
- 路径中含有 `'use client'` 的组件不得作为 `generateMetadata`、`generateStaticParams` 的宿主。
