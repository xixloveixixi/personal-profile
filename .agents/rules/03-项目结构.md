# 项目结构（NON-NEGOTIABLE）

## 顶层目录

```
personal-profile/
├── app/             # Next.js App Router 路由 + Route Handlers
├── components/      # React 组件，按业务域拆分
├── content/         # 本地内容（about、blog、projects 的 JSON / MDX 占位）
├── lib/             # 工具、AI、内容封装、Zustand store
├── public/          # 静态资源
├── scripts/         # 一次性脚本（知识库初始化、掘金发布等）
├── specs/           # PRD / Spec
├── tech_design/     # SDD 与技术设计文档
├── docs/            # 包含 dev-harness 研发流程文档
├── supabase/        # Supabase SQL / 函数
└── 根级文件          # next.config.js、tailwind.config.ts、tsconfig.json 等
```

**禁止在仓库根目录新建 `src/` 顶层目录**。

## `app/` 路由约定

- 公开路由：`/`、`/about`、`/portfolio`、`/portfolio/[slug]`、`/blog`、`/blog/[id]`。
- 私有路由（owner，需登录）：`/dashboard`、`/profile`、`/goals`、`/plans`、`/progress`、`/review/weekly`。
- 后台管理：V1 仅 Go 接口，无前端页面。
- Route Handlers：`app/api/<feature>/route.ts`，公开走 `app/api/public/*`，私有走 `app/api/private/*`。
- 详细约定见 `06-路由规范.md`。

## `components/` 分层

```
components/
├── ui/          # shadcn 原子组件，跨业务域复用
├── shared/      # 业务无关的通用组件（如 ErrorBoundary、Loading）
├── layout/      # 全局布局，如 Header、Footer、LearningWorkspaceLayout
├── chat/        # AI 聊天相关
├── about/       # 关于页专用
├── portfolio/   # 项目页专用
├── blog/        # 博客页专用
├── magicui/     # 视觉特效组件
├── background/  # 背景动效
└── icons/       # 图标
```

判断规则：

- 多页复用且业务无关 → `components/ui` 或 `components/shared`。
- 仅单业务域使用 → `components/<domain>/`。
- 全局布局 / 工作台框架 → `components/layout/`。
- 详细规范见 `04-组件规范.md`。

## `lib/` 分层

```
lib/
├── ai/          # Agent 核心、工具、知识库检索
├── content/     # 本地内容读取（projects、about）
├── constants/   # 业务常量、枚举
├── stores/      # Zustand store，按业务模块拆分
├── utils/       # 通用工具函数
├── notion.ts    # Notion 客户端封装
└── utils.ts     # 兜底通用工具（cn 等）
```

约束：

- 所有 Zustand store 必须放在 `lib/stores/<feature>.ts`，禁止散落在 `app/` 或 `components/` 内。
- Notion / Supabase / Go API 调用统一在 `lib/` 下封装，组件不直接 fetch 第三方。
- 类型定义就近放在使用它们的 `lib/<feature>.ts` 或 `app/api/<feature>/route.ts` 旁；跨域共享类型可放在 `lib/<feature>/types.ts`。

## `content/` 使用约束

- `content/about/*.json`、`content/projects/*.json` 为迁移到 Go 后端前的过渡数据源。
- 按 SDD 5.3，公开个人信息、技能、联系方式、项目最终来源是 Go API；当前阶段仍可读 `content/`，迁移后改为 Route Handler 调用 Go。
- 博客只读 Notion，禁止把博客正文落到 `content/blog/`。

## Mock 数据策略

后端接口未完成时：

- 在 `app/api/<feature>/route.ts` 内返回 mock JSON，类型用 `zod` schema 定义。
- 不创建独立 Zustand store 仅为承载 mock。
- UI 还原度优先于功能完整。
- 禁止预留过多扩展字段或复杂 mock 结构，等待 Go 接口落地后切换。
