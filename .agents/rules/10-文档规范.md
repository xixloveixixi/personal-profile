# 文档规范

## 注释原则

- 使用 JSDoc 风格注释（`/** */`），中文撰写。
- 注释解释「为什么」而不是「是什么」，避免与代码重复。

## 注释类型

| 类型 | 内容 |
|------|------|
| 文件头部 | 说明文件用途与主要职责 |
| 函数 / 方法 | 说明用途、参数、返回值、副作用 |
| 复杂逻辑 | 解释关键算法、业务规则或特殊处理 |
| 类型定义 | 说明接口、类型、枚举的含义与用途 |
| Route Handler | 顶部注明数据源（Notion / Supabase / Go API / LLM）与权限（公开 / 私有） |

## 内容文件

- `content/about/*.json`、`content/projects/*.json`、`content/blog/**` 内的纯内容文件不写代码注释；如需说明字段，在 `lib/content/<feature>.ts` 类型定义旁注释。
- Notion 拉取的博客内容只读，不在前端添加二次注释。

## SDD 与 Harness 联动

- 重大设计决策、表结构、接口边界以 `tech_design/ai-learning-platform/SDD-AI学习规划与成长记录平台.md` 为准。
- 与代码相关的临时决策、踩坑记录到 `docs/dev-harness/progress-log.md` 或 `pitfalls.md`，不要散落在代码注释中。
