# Spec-Gap 差异分析

## 触发条件

用户验收时反馈"不对"、"不符合预期"、"有问题"、"不是这个效果"。

**快路径**（不走 spec-gap，直接修复）：
- 拼写错误、文案调整
- 样式轻微偏差
- 字段名/标签显示问题

## 输出格式

```
[FSM: SPEC_GAP]

用户原话：[用户反馈]

差异定位：
  预期行为：[用户最终想要的]
  实际行为：[当前实现]

可能根因：

□ 需求阶段：范围遗漏或理解偏差
   证据：[如"Gate A 确认中未提及此场景"]
   回退目标：REQUIREMENT

□ 设计阶段：接口/表结构与需求不匹配
   证据：[如"api-contract.md 缺少此接口"]
   回退目标：DESIGN

□ 编码阶段：实现偏差
   证据：[如"代码逻辑与契约一致但实现有 bug"]
   回退目标：CODING

建议回退：[REQUIREMENT / DESIGN / CODING]
理由：[一句话总结]
涉及文件：[可能需要修改的文件列表]

请确认是否同意此分析，或补充信息。
```

## 回退规则

- AI **不自动删除或 revert** 任何代码
- 仅输出受影响文件清单，标记"可能需重做"
- 用户决定保留或删除

回退话术示例（回退到 DESIGN）：

```
[FSM: DESIGN ← Spec-Gap 回退]

根据分析，需要在 api-contract.md 中补充 [遗漏项]。
完成后回复"确认冻结"重新进入编码。

以下文件可能需重做：[文件列表]
请决定保留或删除。
```

## 回退路径

```
ACCEPTANCE → 用户说"不对"
    ↓
SPEC_GAP 分析
    ↓
├── 需求问题 → REQUIREMENT → 重新确认范围 → DESIGN → ...
├── 设计问题 → DESIGN → 重新冻结契约 → CODING → ...
└── 编码问题 → CODING → 修复代码 → GUARD_CHECK → ...
```
