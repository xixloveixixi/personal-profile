# Workflow 自检规则

WORKFLOW_REVIEW 阶段的检查逻辑和修改权限。

## 检查项

| 信号 | 阈值 | 含义 |
|------|------|------|
| 状态卡顿 | 同一状态停留 >3 轮对话 | 状态定义不清晰或离开条件过严 |
| 频繁回退 | 同一阶段 spec-gap 回退 >2 次 | 前置检查不足或需求/设计环节有缺陷 |
| 重复门禁失败 | 相同错误类型 >2 次 | AUTO_FIX 覆盖不全或门禁顺序需调整 |
| 用户跳过 | 用户要求跳过某环节 >1 次 | 该环节可能应标记为可选 |
| 快路径滥用 | 快路径修复 >3 次/阶段 | 验收标准可能需细化 |

## 修改权限

WORKFLOW_REVIEW 是**自动状态**，AI 可以直接修改以下文件：

| 文件 | 允许修改 | 示例 |
|------|---------|------|
| `skill.md` | 阈值、状态转换条件 | 把"2 轮不收敛"改为"3 轮" |
| `references/guard-gates.md` | 软硬门禁分类、修复 skill 映射 | 新增一个软门禁检查项 |
| `references/spec-gap.md` | 回退规则、根因分类 | 细化"编码阶段问题"的子分类 |
| `references/error-handling.md` | 降级策略 | 新增一种文件缺失的处理方式 |

**不允许修改**：
- FSM 状态图的核心结构（增删状态需用户确认）
- 用户确认权限（不能把"用户驱动"改为"AI 驱动"）
- AGENTS.md 中的硬性规则

## 修改流程

```
1. 识别问题信号
2. 分析根因
3. 提出修改方案（内部决策，不询问用户）
4. 执行修改
5. 记录到 progress-log.md：
   - 修改了什么
   - 为什么修改
   - 预期效果
6. 静默结束
```

## 记录格式

```markdown
### WORKFLOW_REVIEW 自动修改

**日期**：YYYY-MM-DD
**触发信号**：[描述观察到的问题]
**根因分析**：[为什么会出现这个问题]
**修改内容**：
- 文件：xxx.md
- 变更：[具体修改]
**预期效果**：[下次流程执行时应该如何改善]
```

## 示例

### 示例 1：门禁收敛阈值调整

**信号**：本次流程 gofmt 软门禁连续失败 3 次才收敛

**分析**：某些文件格式问题需要多轮修复，2 轮阈值过严

**修改**：
```diff
- 连续 2 轮相同错误 → HARD_FAIL
+ 连续 3 轮相同错误 → HARD_FAIL
```

### 示例 2：新增软门禁

**信号**：本次流程 `go vet` 警告导致后续问题

**分析**：`go vet` 应纳入软门禁，而非等到硬门禁失败才发现

**修改**：`references/guard-gates.md` 软门禁列表新增 `go vet`
